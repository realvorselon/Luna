# Luna Visual Direction: Moonlit Garden Return

## Core visual north star

Luna should feel like a quiet nighttime return place, not a dashboard. The world is moonlit, soft, clean, comforting, and lightly Japanese-inspired.

The main metaphor is: moonlit sky → garden gate → lantern/path → gentle return.

The opening should feel like a “press start” threshold, not a project summary page. Luna should help the user feel good about coming back and gently satisfied after taking one step.

## Opening screen direction

The first screen should be mostly atmosphere and invitation. Most project details should stay hidden until after the user enters. The moon, stars, and Luna name establish presence.

A torii-inspired or garden-gate-inspired shape can become the primary button itself. The gate button should say “Return gently” or equivalent. Pressing the gate means crossing the threshold back into the project.

Secondary actions like “Shape this return” or “Open full overview” should stay visually quiet. Avoid making the opening feel like an admin page, form, or project summary.

## Luna’s presence

Luna is primarily represented through the moon and nighttime atmosphere. Small night companions can suggest her presence when useful: fireflies, a gentle moth, or similar tiny night-life accents.

These should be cute and comforting, not creepy, mascot-like, childish, or distracting. They should appear only when they serve guidance, such as softly drawing attention to the next step after stillness.

Luna should feel present through atmosphere, guidance, and care rather than constant character art.

## UI graphics scale

Luna should remain a usable app UI, not a full illustrated game scene. Garden graphics should be small-ish accents beside or near content.

Motifs can include gate, moon, lantern, path stones, flowers, fireflies, moth, plaques, shrine slips, and garden markers. These accents should support readability and direction, not overwhelm the app.

The garden should feel like a visual language for the UI, not a background painting that fights the content.

## Game-design logic without gamification

Luna can borrow from video game design principles: framing, contrast, light, color, pathing, gentle feedback, and visual attention cues. Use visual design to answer:

- Where should the user look?
- What is the one gentle action?
- How should returning feel?

Do not turn Luna into a game. No points, XP, streaks, achievements, quests, badges, pressure loops, or completion addiction.

The goal is to help the user know where to look, feel good about returning, and feel gently satisfied after taking a step. Completion feedback should be warm and relieving, not addictive or noisy.

## Guided Return direction

Guided Return should feel like moving along a quiet moonlit path, not filling out a workflow form. The zigzag/path-stone idea can be used as a visual motif for “next step” and progression, but should not become loud, boring, or overly literal.

The One Next Action is the warm lantern. Each Guided Return stage should have one clear focal point. Completion should feel warm and relieving, not gamified. Rest should feel like permission to stop.

## Full overview/editing direction

Full overview/editing should feel like quiet garden markers, plaques, shrine slips, or path objects. It should not feel like admin settings, a Notion clone, or a project-management dashboard.

It may be more practical than the opening screen, but it should still belong to the same moonlit garden world. Editing fields should remain clear and usable, but the surrounding framing should feel gentle and intentional.

## Avoid

- Corporate dashboard
- Notion clone
- Sterile therapy app
- Childish bedtime/nursery look
- Boring plain cards
- Loud neon/casino glow
- Making Luna feel like a project manager
- Constant animation or visual noise
- Culturally overloaded shrine theming
- Using Japanese inspiration as a pile of symbols
- Full-screen decorative art that hurts usability
- Motion or glow that could trigger nausea or eye strain

## Implementation rules

Future visual PRs should preserve current behavior first. Work one visual slice at a time, with no new features while translating the visual identity.

Keep motion calm, rare, slow, and compatible with reduced-motion preferences. Keep mobile comfort and readability as hard constraints. Small accents should guide attention, not entertain or distract.

If a visual idea harms usability, comfort, or clarity, simplify it. Do not implement broad redesigns in one PR. Do not introduce new storage keys or network behavior for visual work.
