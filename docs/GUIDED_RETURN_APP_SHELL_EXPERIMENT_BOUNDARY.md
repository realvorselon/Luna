# Guided Return App Shell Experiment Boundary

This documentation-only boundary sets the next meaningful Guided Return product experiment. It does not implement the shell. It does not change `prototype.html`, `index.html`, CSS, JavaScript, runtime behavior, motion, mobile controls, stage styling, editable fields, localStorage behavior, graphics, assets, backend shape, accounts, database, sync, AI generation, moderation logic, crisis detection, multi-project behavior, task systems, calendars, habit tracking, or Philosophy Garden content.

## What the Existing Guided Return Work Proved

The current Guided Return work is useful and should be preserved as evidence.

- Optional guided mode is technically viable.
- The full overview can remain available and useful as a reference mode.
- Stage navigation works for the limited Orient → Remember → Choose slice.
- Forward and Back navigation work.
- Returning to the full overview works.
- Existing editable values remain preserved.
- One Next Action works as the lantern of the loop and should remain the strongest visual focus.
- Calm motion can help orientation and comfort, especially when it treats the panel as the moving surface.
- Motion alone does not create the feeling of a finished app experience.
- Mobile controls must remain clear, reachable, and understandable.

## Why a New Shell Experiment Is Justified

The current guided view still fundamentally feels like sections of a webpage being hidden and revealed. It is calmer than the first slice, and the recent motion, surface, mobile-control, and entrance experiments made useful improvements, but they have reached diminishing returns for this exact shell.

The next experiment should test the container and spatial experience, not another individual color, shadow, duration, surface tweak, or card adjustment. This is an evolution of Guided Return, not a rejection of the work already completed. The existing implementation remains the proven reference implementation for the limited guided loop.

Do not automatically continue CSS or motion micro-polish. Do not automatically add Set Aside, Record, or Rest merely to complete the sequence.

## App-Shell Experiment Direction

A later implementation should test one focused Guided Return shell that feels closer to a modern app screen while still belonging to Luna's moonlit return path.

Preferred direction:

- The dark moonlit page remains the stable environmental background.
- Guided Return appears as one intentional focused surface.
- On mobile, the focused surface may occupy most of the available viewport.
- On larger screens, the focused surface may appear as a spacious centered app panel.
- Stage content changes within that stable shell.
- Navigation stays in a predictable action area.
- The shell reduces the feeling of ordinary webpage scrolling.
- The full overview remains accessible as a separate reference mode.
- One Next Action remains visually strongest.
- Orient and Remember may share a quieter related surface language.
- The shell should feel like entering a calm place that remembers where the person left off.

## Scope for the First Later Implementation

The first later implementation should test only the shell and spatial experience.

It should:

- keep only Orient → Remember → Choose
- reuse existing content and editable values
- preserve all localStorage keys and behavior
- preserve reduced-motion support
- preserve Return to full overview
- preserve the current prototype as the proven reference implementation
- preserve the full overview as the reference mode
- avoid adding Set Aside, Record, or Rest
- avoid adding graphics yet
- avoid adding backend or broader app architecture
- avoid changing the underlying Luna loop

## What the Shell Must Not Become

The shell must not become:

- a modal trap
- a wizard or onboarding funnel
- a carousel
- a productivity dashboard
- a full-screen task manager
- a collection of nested cards
- a rigid fixed-height surface that clips content
- an internally scrolling form unless genuinely unavoidable
- Jira with moon stickers

Emperor Jira does not receive a key to the new shell.

## Success Question

The later shell experiment succeeds only if human testing says Guided Return feels more like entering and moving through one calm focused app space, and less like scrolling through or swapping sections of a webpage.

If the shell only looks like another styled webpage section, it has not answered the right question.

## Do Not Implement in This Run

Do not:

- change `prototype.html`
- change `index.html`
- change CSS or JavaScript
- build the app shell
- change current motion
- change mobile controls
- change stage styling
- add stages
- add graphics or assets
- change editable fields
- change persistence
- add backend, accounts, database, sync, AI generation, moderation logic, crisis detection, multi-project behavior, task systems, calendars, habit tracking, or new Philosophy Garden content

Human-review this app-shell boundary before implementing one bounded shell experiment for only Orient → Remember → Choose.
