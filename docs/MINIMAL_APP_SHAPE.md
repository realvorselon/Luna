# Minimal App Shape Review

This note reviews the smallest possible future Luna app shape suggested by `docs/APP_SEED.md`. It is only a calm planning note, not a build plan, roadmap, feature list, or promise.

## What the Smallest Usable App Should Help With

The smallest usable Luna app should help one person return to one small project without feeling lost. It should help them remember what the project is, notice the current state, see what changed last, choose or skip one kind next action, and stop before adding too much.

## What the First Version Should Avoid

The first version should avoid managing many projects, adding complex features, asking for private details, creating pressure to be productive, or pretending the whole project can be solved at once. It should avoid accounts, pricing, dashboards full of options, technical setup choices, and anything that turns Luna into a command center.

## Simplest User Flow

1. Create or open one small project.
2. Answer a gentle setup question.
3. See current state, current mode, what changed last, and one next action.
4. Do or skip the next action.
5. Record what changed.
6. Rest before adding more.

## Smallest Project State to Remember

Luna would only need to remember the project's plain-language name or focus, the gentle setup answer, the current state, the current mode, the last recorded change, and one possible next action. If something is not needed to help the person return and continue calmly, it can wait.

## How Stewardship Modes Could Support This Later

The existing stewardship modes could help Luna name the kind of support the project needs without making the user choose from a large menu. Orient could help with returning, Grow could support one small addition, Align could keep the record clear, Prune could reduce clutter, Reflect could check whether the next step still fits, and Rest could protect the pause after useful movement.

## Why This Should Stay Separate From Building

This review should stay separate from building so Luna can understand the app shape before any app feature exists. Keeping it as a note protects the one-small-change rhythm, avoids premature architecture, and leaves the next step simple: rest before adding any app feature, dashboard feature, visual styling, backend, account system, roadmap expansion, or additional planning note.
