# Guided Return Motion Experiment Boundary

This documentation-only boundary defines the next small Guided Return experiment after the first Orient → Remember → Choose slice passed its initial human technical and visual tests. It does not change `prototype.html`, `index.html`, CSS, JavaScript, runtime behavior, guided stages, navigation controls, editable fields, localStorage behavior, blank-value fallback behavior, clear-local-edits behavior or scope, backend shape, accounts, database, sync, AI generation, moderation logic, crisis detection, multi-project behavior, task lists, checklists, kanban, workflow engines, calendars, habit tracking, or Philosophy Garden content.

## What the first slice proved

The first Guided Return slice proved that Luna can offer a small, optional guided path without replacing the full overview. A person can begin the guided return, move forward through Orient → Remember → Choose, move backward where a previous stage exists, return to the full overview at any time, and keep existing editable values intact.

The follow-up button-depth polish also proved that the guided controls can read clearly enough as controls while remaining calm, moonlit, and consistent with the prototype direction.

## Why a motion boundary is useful

The next question is not whether Luna should complete the whole six-stage sequence. The next question is whether a very small motion or transition cue would make the already-proven guided slice feel more like a calm path and less like a form, without making it feel busy, gamified, or procedural.

Motion should only support orientation. It should not become decoration, reward, urgency, productivity pressure, or a new interaction model.

## Experiment scope

A later implementation may test one tiny motion treatment around the existing guided view only, such as a restrained stage-change cue or a subtle sense that the current guided card has changed.

If implemented later, the experiment should:

- stay limited to the existing Orient → Remember → Choose guided slice
- preserve the full overview as the proven reference mode
- preserve existing guided controls and navigation behavior
- preserve existing editable values and localStorage keys
- preserve the clear-local-edits behavior and scope
- remain calm, optional, and reversible
- respect reduced-motion preferences if any motion is added
- avoid adding Set Aside, Record, or Rest merely to complete the sequence

## Do not implement in this boundary run

Do not change `prototype.html` or `index.html` in this boundary run.

Do not add or alter CSS, JavaScript, animations, transitions, graphics, icons, visual assets, guided stages, navigation rules, editable fields, persistence, localStorage keys, blank-value fallback behavior, clear-local-edits behavior, backend shape, accounts, database, sync, AI generation, moderation logic, crisis detection, multi-project behavior, task lists, checklists, kanban, workflow engines, calendars, habit tracking, or Philosophy Garden content.

## Success criteria for a later motion experiment

A later motion experiment would succeed only if it makes the existing guided slice feel a little easier to follow while staying quiet enough that Luna still feels like a place to return and rest.

It should fail or be removed if it makes the guided view feel flashy, game-like, slower, more procedural, less accessible, or more important than the full overview.

## Next suggested step

Human-review this motion boundary before deciding whether one tiny guided-stage motion experiment should proceed. Keep the first Guided Return read-through intact as the recorded proof of what the first slice already showed.
