# First Editable Field Boundary

This note defines Luna's first tiny editable boundary for a later run. It is only a boundary note, not the editable UI implementation.

## First Editable Fields

The first editable fields should be only:

- project name
- current goal

These two fields are enough to help someone make the one-project prototype feel personally useful without turning Luna into a task manager, backlog, form, or project management system.

## Editing Feeling

Editing should later feel calm, reversible, and small. The first editable version should let a person gently rename the project and adjust the current goal without making every card editable at once.

The first editable version should not edit every card at once.

## Keep Static Until Later Boundaries

These areas should remain static until later boundaries are defined:

- Current Mode
- last completed run
- Next Action
- Ignore For Now
- Record the Change
- rest / next step

Keeping these areas static protects Luna's return loop from becoming a task manager before the smallest useful editable surface is understood.

## Preserve the Reversible Preview Path

The prototype should preserve the dashboard ↔ prototype reversible path, including the “Return to dashboard” link. A visitor should still be able to move from the public dashboard into the prototype preview and back again without feeling trapped in a new app surface.

## Not Included

This note does not add editable UI, forms, inputs, textareas, selects, buttons, save actions, local storage, backend, accounts, database, dashboard expansion, visual redesign, mode workflow, multi-project system, or new Philosophy Garden content.
