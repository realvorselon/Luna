# Guided Return Phase Boundary

This documentation-only boundary defines the next Luna phase after the completed single-project browser prototype phase. It does not change `prototype.html`, `index.html`, CSS, JavaScript, runtime behavior, navigation controls, back / forward controls, mode switching, transitions, animations, graphics, visual assets, editable fields, localStorage behavior, blank-value fallback behavior, clear-local-edits behavior or scope, backend shape, accounts, database, sync, AI generation, moderation logic, crisis detection, multi-project behavior, task lists, checklists, workflow engines, or Philosophy Garden content.

The current single-project browser prototype phase remains complete and closed. It proved the core Luna loop: someone can return to one unfinished project, see where they are, remember what changed last, identify one clear Next Action, know what can wait, record the change, and rest.

## Working phase direction

The next phase direction is **Guided Return**.

Guided Return is a small exploration of the experience of returning to one unfinished project. It asks whether Luna can feel less like a form and more like a calm path someone moves through, while preserving the proven one-project return loop.

## What this phase is trying to prove

The Guided Return phase is trying to prove that Luna can offer a small, reversible guided experience through part of the existing return loop without making the app feel more procedural, more productive, or more complicated.

The conceptual path remains:

1. Orient
2. Remember
3. Choose
4. Set Aside
5. Record
6. Rest

A guided path mode may let someone move through that loop one stage or decision at a time. Moving forward should feel calm and intentional, not like completing a wizard, survey, onboarding funnel, checklist, or productivity workflow. A person should be able to move backward when useful. The path should not create pressure to complete every stage before being allowed to rest.

## What must remain true from the completed prototype phase

The completed prototype phase protects the core Luna loop. Guided Return must preserve these truths:

- Luna stays focused on one unfinished project.
- The existing core state remains protected: project identity, Current Mode, current goal, what changed last, one Next Action, Ignore For Now, Record the Change, and Rest.
- One Next Action remains the lantern of the loop.
- Ignore For Now does not become a backlog.
- Rest is treated as permission or state, not merely another text field.
- Every section should not become editable by default.
- Local-only saving limits, blank-value fallbacks, and clear-local-edits behavior should not be changed without a later explicit boundary.
- The current full-page prototype remains a useful proven reference.

## Full overview mode

Guided path mode should complement the overview rather than destroy or replace it.

Preserve a way to see the whole one-project state at once. The current full-page prototype is still valuable because it shows the complete one-project state in one place. Switching between guided and overview modes may be explored later, but it should not be implemented in this boundary run.

## Experience direction

Luna should feel like returning to a quiet place that remembers where you left off.

The finished visual direction remains moonlit garden, quiet desk, and gentle return path. Later restrained transitions or graphics may support the experience, including subtle scroll-triggered or stage-triggered visual arrivals. Actions should feel like they arrive or settle rather than explode, celebrate productivity metrics, or demand attention.

No transitions, animations, graphics, or new visual assets should be implemented in this boundary run.

## Fresh-user clarity

Preserve the observation from the prototype completion review that completely fresh users may need a clearer introduction to Luna's purpose than the current recipe / soup placeholder language provides.

Do not change prototype copy in this run. Do not assume the sample recipe project is the finished Luna theme.

## What would count as success

The Guided Return phase succeeds if Luna can demonstrate one small, reversible guided experience through part of the existing return loop while preserving access to the full overview, keeping one project in focus, and making returning feel calmer rather than more procedural.

Success does not require implementing the entire Orient → Remember → Choose → Set Aside → Record → Rest journey. Success also does not require animations, graphics, new visual assets, mode switching, new persistence behavior, or broader app architecture.

## Explicitly outside this phase

This phase is not:

- a multi-project manager
- a task list
- a checklist system
- a kanban board
- a workflow engine
- a Jira-like productivity system
- a calendar
- a habit tracker
- an AI project generator
- a redesign for its own sake

Do not turn Luna into Jira with moon stickers.

## Do not implement in this boundary run

Do not:

- change `prototype.html`
- change `index.html`
- change CSS or JavaScript
- change runtime behavior
- add navigation controls
- add back / forward controls
- add mode switching
- add transitions or animations
- add graphics or visual assets
- add or modify editable fields
- change localStorage behavior
- change blank-value fallback behavior
- change clear-local-edits behavior or scope
- add backend, accounts, database, sync, AI generation, moderation logic, crisis detection, multi-project behavior, task lists, checklists, workflow engines, or new Philosophy Garden content

## First implementation slice direction

After this boundary is reviewed, consider one small reversible guided-view slice that tests only part of the existing return loop. The first slice should prove the interaction model before visual flourish.

Do not implement the entire Orient → Remember → Choose → Set Aside → Record → Rest journey at once. Do not add animations or graphics yet. Do not design the entire finished app before testing whether one calm guided step can coexist with the full overview.

Human-review this Guided Return phase boundary and decide whether its first implementation slice should proceed.
